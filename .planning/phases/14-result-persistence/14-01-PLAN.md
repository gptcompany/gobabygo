# Plan 14-01: Result Persistence + Read Path

## Phase Reference

- **Phase**: 14 — Result Persistence + Read Path
- **Goal**: Il router persiste i result che i worker gia' inviano, e li rende leggibili via API
- **Context**: `.planning/phases/14-result-persistence/14-CONTEXT.md`
- **Brief**: `CROSS_VERIFICATION_BRIEF.md` sezione 7.2, Fase 14

## Overview

Single plan che copre tutti i deliverable della Fase 14:
1. Task model + DB migration (result field)
2. Server-side result extraction + scheduler persistence
3. GET /tasks/{id} e GET /tasks?status=... endpoints
4. Result truncation + secret filtering
5. Test completi

## Tasks

### Task 1: DB Migration + Model Update

**Files**: `src/router/db.py`, `src/router/models.py`

**db.py changes:**

1. In `init_schema()`, aggiungere migration backward-compatible:
```python
# After CREATE TABLE tasks (existing)
# Migration: add result_json column if missing
conn.execute("""
    ALTER TABLE tasks ADD COLUMN result_json TEXT DEFAULT NULL
""")
```
Wrapped in try/except (column may already exist on re-run).

2. Aggiungere helper `_sanitize_result()`:
- Input: `dict | None`
- Filtra pattern secrets: `sk-[a-zA-Z0-9]{20,}`, `ghp_[a-zA-Z0-9]{36,}`, `xoxb-[a-zA-Z0-9-]{50,}`
- Se JSON serializzato > 32KB: tronca value dei campi `output`/`final_snapshot` + aggiungi `_truncated: true`
- Return: JSON string

3. Update `get_task()` per leggere `result_json` e deserializzare in `result` field.

4. Aggiungere metodo `list_tasks(status: str | None = None, limit: int = 100) -> list[Task]`:
- Query con filtro opzionale per status
- Limit default 100
- Ordine: created_at DESC

**models.py changes:**

5. Aggiungere a `Task`:
```python
result: dict[str, Any] | None = None
```

**Tests**: Inline con Task 4.

### Task 2: Scheduler Result Persistence

**Files**: `src/router/scheduler.py`

1. Aggiornare signature di `complete_task()`:
```python
def complete_task(self, task_id: str, worker_id: str, result: dict[str, Any] | None = None) -> bool:
```

2. Passare `result` a `_route_to_review()` e `_route_to_completed()`:
```python
def _route_to_review(self, task: Task, worker_id: str, result: dict[str, Any] | None = None) -> bool:
def _route_to_completed(self, task: Task, worker_id: str, result: dict[str, Any] | None = None) -> bool:
```

3. In ENTRAMBI i metodi, aggiungere nella transazione (dopo update_task_status, prima di expire_lease):
```python
if result is not None:
    sanitized = self._db._sanitize_result(result)
    self._db.update_task_fields(
        task.task_id,
        {"result_json": sanitized},
        conn=conn,
    )
```

**Tests**: Inline con Task 4.

### Task 3: Server Handler + New Endpoints

**Files**: `src/router/server.py`

1. **Update `_handle_task_complete()`**: Estrarre result dal body e passare allo scheduler:
```python
result = data.get("result")  # None se non presente (backward compat)
ok = scheduler.complete_task(task_id, worker_id, result=result)
```

2. **Nuovo handler `_handle_get_task()`** per `GET /tasks/{id}`:
- Estrarre task_id dal path (regex: `/tasks/([^/]+)$`)
- Chiamare `db.get_task(task_id)`
- Se None: 404
- Se esiste: 200 con task.model_dump()

3. **Nuovo handler `_handle_list_tasks()`** per `GET /tasks?status=...`:
- Estrarre query params: `status` (opzionale), `limit` (default 100)
- Chiamare `db.list_tasks(status=status, limit=limit)`
- 200 con lista di task

4. **Registrare routes** nel routing table:
- GET `/tasks` → `_handle_list_tasks` (esatto, no trailing slash)
- GET `/tasks/<id>` → `_handle_get_task` (pattern match)
- Attenzione: GET /tasks/next gia' esiste — la route deve avere priorita' (match esatto prima di pattern)

**Tests**: Inline con Task 4.

### Task 4: Test Suite

**File**: `tests/test_result_capture.py` (nuovo)

Test da implementare (~120 LOC):

1. `test_complete_with_result_persists` — POST /tasks/complete con result → result in DB
2. `test_complete_without_result_backward_compat` — POST /tasks/complete senza result → OK, result NULL
3. `test_result_persisted_on_review_transition` — task critico: running → review con result
4. `test_result_truncation_32kb` — result > 32KB → troncato con _truncated flag
5. `test_result_secret_filtering` — result con sk-xxx viene sanitizzato
6. `test_get_task_by_id` — GET /tasks/{id} ritorna task con result
7. `test_get_task_not_found` — GET /tasks/{id} con ID inesistente → 404
8. `test_list_tasks_all` — GET /tasks ritorna lista
9. `test_list_tasks_by_status` — GET /tasks?status=completed → solo task completati
10. `test_list_tasks_empty` — GET /tasks?status=xxx → lista vuota

Pattern: usa conftest.py esistente per setup router/DB in-memory.

## Execution Order

Task 1 → Task 2 → Task 3 → Task 4

Task 1 e' prerequisito per tutti (model + DB).
Task 2 dipende da Task 1 (scheduler usa DB).
Task 3 dipende da Task 2 (handler chiama scheduler).
Task 4 dipende da tutti (test end-to-end).

## Success Criteria

1. `POST /tasks/complete {"task_id":"...","worker_id":"...","result":{...}}` persiste result in DB
2. `POST /tasks/complete` senza result continua a funzionare (backward compatible)
3. Result persistito anche su transition a `review` (task critici) — stessa transazione
4. `GET /tasks/{id}` ritorna task completo con result
5. `GET /tasks?status=completed` lista task filtrati per status
6. Result > 32KB viene troncato con flag `_truncated: true`
7. Secret patterns filtrati prima di persistenza
8. Tutti i test passano

## Breaking Changes

None. Tutti i cambiamenti sono additivi:
- Campo `result` opzionale con default None
- Colonna `result_json` aggiunta con DEFAULT NULL
- Nuovi endpoint GET non conflittano con esistenti
- `complete_task()` signature backward-compatible (result=None di default)
