# Phase 19: Approval + Notification Bridge

## Phase Reference

- **Phase**: 19 — Approval + Notification Bridge
- **Goal**: Emit useful Matrix notifications for human intervention without changing the router critical path
- **Context**: `.planning/milestones/v1.4-PHASE-19-BREAKDOWN.md`

## Scope

Implement the MVP Matrix bridge as an out-of-process notification-plane component.

This phase:
- polls existing router HTTP endpoints only
- consumes `Session.metadata.attach_kind` / `attach_target`
- emits Matrix notifications for the MVP triggers
- keeps all bridge state local and rebuildable

This phase does **not**:
- add direct DB reads
- replace terminal attach
- become a second source of truth
- change router scheduling, task routing, or FSM behavior

## Delivery Slices

### 19A — Bridge Skeleton + Config

Create a standalone bridge process/script with env-based config.

Required config:
- `MESH_ROUTER_URL`
- `MESH_AUTH_TOKEN`
- `MESH_MATRIX_HOMESERVER`
- `MESH_MATRIX_ACCESS_TOKEN`
- `MESH_MATRIX_DEFAULT_ROOM`
- `MESH_MATRIX_POLL_INTERVAL_S`

Optional config:
- `MESH_MATRIX_BOSS_ROOM`
- `MESH_MATRIX_INPUT_PATTERNS`
- `MESH_MATRIX_REQUEST_TIMEOUT_S`

Primary deliverable:
- new process entrypoint under `scripts/` or a small dedicated module outside `src/router/`

### 19B — Router Polling + Local State

Read only from existing endpoints:
- `GET /sessions?state=open`
- `GET /sessions/messages?session_id=...&after_seq=N`
- `GET /tasks?status=review`
- `GET /threads`

Track only rebuildable local state:
- open sessions seen
- last `seq` per `session_id`
- last review task ids seen
- last thread status per `thread_id`

No DB writes and no router state mutation.

### 19C — Trigger Detection

Implement MVP triggers:

1. `input_requested`
   - match new outbound CLI text against configured patterns
   - default patterns should cover:
     - `approve`
     - `continue`
     - `press enter`
     - `y/n`
     - `select`

2. `approval_needed`
   - detect task entering `review`

3. `thread_failed`
   - emit once per status transition

4. `thread_completed`
   - emit once per status transition

`thread_blocked` remains optional in the first code pass if the existing thread API does not expose it cleanly enough; do not expand scope to force it.

### 19D — Matrix Rendering + Send

Each notification must include, when available:
- repo
- session id
- task id
- thread id
- trigger / reason
- `attach_kind`
- `attach_target`
- rendered attach command

Rendering rules:
- `upterm`: render a human copyable SSH command from `ssh://TOKEN@HOST:PORT`
- `ssh_tmux`: render an SSH command plus target tmux session name
- if no attach is available, state that clearly

Notification copy must explicitly say:
- quick text reply is only suitable for simple input
- full control requires terminal attach

## Files

Preferred files to add or modify:
- new `scripts/mesh-matrix-bridge.py` (preferred)
- optional `deploy/` runbook or service note for `muletto`
- optional small helper in `src/meshctl.py` only if it materially reduces operator friction

Files to avoid:
- `src/router/db.py`
- `src/router/fsm.py`
- `src/router/scheduler.py`
- `src/router/dependency.py`
- `src/router/verifier.py`

Avoid router changes unless an existing read endpoint is provably insufficient.

## Acceptance Criteria

1. A live interactive session that requests input produces a Matrix notification
2. A task entering `review` produces a single `approval_needed` notification
3. A thread transition to `failed` or `completed` produces a single useful notification
4. The notification includes the correct attach handle when one exists
5. Restarting the bridge does not affect task execution and only rebuilds local notification state

## Test Strategy

### Unit Tests

- trigger matching for `input_requested`
- attach target rendering for:
  - `upterm`
  - `ssh_tmux`
  - no attach
- de-duplication behavior for review/thread transitions

### Integration Tests

- fake router responses -> bridge emits expected Matrix payloads
- bridge restart -> local state rebuilds without crashing

### Manual Validation

- run bridge against a live router
- create an interactive session
- inject a prompt requiring human input
- verify Matrix message content and attach rendering

## Execution Order

1. Build bridge skeleton + config parsing
2. Add router polling and local state
3. Implement trigger detection
4. Implement Matrix rendering/sending
5. Add tests
6. Validate manually against one real interactive session

## Rollout Notes

- Deploy first on `muletto`
- Start with one room and one operator
- Keep the bridge disabled by default until manual validation passes
- Do not block `Phase 20` planning on `thread_blocked` if the MVP already covers `input_requested`, `approval_needed`, `thread_failed`, and `thread_completed`
