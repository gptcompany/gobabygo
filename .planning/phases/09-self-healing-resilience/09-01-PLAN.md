Modify the `heartbeat_loop()` inner function in `MeshWorker._start_heartbeat()` (worker_client.py lines 98-109) to parse the heartbeat response body and trigger re-registration on `unknown_worker`.

**Changes to `heartbeat_loop()`:**

1. Capture the response from `self._session.post(...)` instead of discarding it:
```python
resp = self._session.post(
    url,
    json={"worker_id": self.config.worker_id},
    timeout=3,
)
```

2. After the POST, parse the response JSON and check the `status` field. Wrap in try/except for backward compatibility with old servers that may not return JSON:
```python
try:
    data = resp.json()
    if data.get("status") == "unknown_worker":
        logger.warning(
            "Router reports unknown_worker for %s, re-registering",
            self.config.worker_id,
        )
        try:
            self._register()
        except requests.RequestException as re_err:
            logger.error("Re-registration failed: %s", re_err)
except (ValueError, KeyError):
    pass  # Old server or non-JSON response -- ignore
```

3. The `except requests.RequestException` block stays unchanged (heartbeat network failure handling).

4. Important: the `_register()` call is already idempotent (POST /register returns 200 for re-registration, 201 for new). No additional guards needed.

5. **Critical**: the `_register()` call MUST be wrapped in its own try/except inside the heartbeat loop. A transient network error during re-registration must NOT kill the heartbeat thread — it should log the error and retry on the next heartbeat cycle.

Add a periodic daemon thread in `run_server()` (server.py) that calls `VerifierGate.check_review_timeout(db)` at a configurable interval.

**Changes to `server.py`:**

1. Add import at top: `from src.router.verifier import VerifierGate`

2. In `run_server()`, after creating the `metrics` instance (line 382) and before `start_time`:

Read config:
```python
review_check_interval = float(os.environ.get("MESH_REVIEW_CHECK_INTERVAL_S", "60"))
```

Create VerifierGate and start review check thread:
```python
verifier_gate = VerifierGate()

def review_check_loop() -> None:
    """Periodically check for stale reviews and transition them to timeout."""
    while True:
        time.sleep(review_check_interval)
        try:
            timed_out = verifier_gate.check_review_timeout(db)
            if timed_out:
                logger.info("Review timeout: %d tasks timed out: %s", len(timed_out), timed_out)
        except Exception as e:
            logger.error("Review timeout check failed: %s", e)

review_thread = threading.Thread(target=review_check_loop, daemon=True, name="review-check")
review_thread.start()
```

3. Store `verifier_gate` in `router_state` for testability:
```python
"verifier_gate": verifier_gate,
"review_check_interval": review_check_interval,
```

**Pattern notes:** This follows the same daemon thread pattern as the existing `watchdog_loop()` (server.py lines 415-421). The thread is daemon=True so it dies with the server. The `time.sleep` is FIRST (before check) so the loop doesn't fire immediately on startup -- same as how the existing watchdog works.
