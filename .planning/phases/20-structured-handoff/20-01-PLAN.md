# Phase 20: Structured Cross-Repo Handoff

## Phase Reference

- **Phase**: 20 — Structured Cross-Repo Handoff
- **Goal**: Replace copy/paste with a structured handoff object while reusing the existing thread model
- **Context**: `.planning/milestones/v1.4-phases/20-structured-cross-repo-handoff/CONTEXT.md`

## Scope

Implement a structured handoff packet as a nested object inside `Task.payload` for cross-repo thread steps.

This phase:
- adds a `HandoffPacket` Pydantic model for validation
- validates handoff structure in `add_step()` when `payload.handoff` is present
- enforces `PRESIDENT_GLOBAL` role for cross-repo handoffs (hard block, 403)
- validates repo names against topology when available
- adds `[HANDOFF]` marker in `meshctl thread status`
- adds `meshctl thread handoff` subcommand for detailed inspection
- adds unit and integration tests

This phase does **not**:
- add new DB columns or schema changes
- change FSM, scheduler, or dependency resolution logic
- change how `thread_context` is computed or propagated
- add new event types to the event bridge
- change the session bus or any transport layer

## Delivery Slices

### 20A — HandoffPacket Model + Validation

Add `HandoffPacket` Pydantic model in `src/router/models.py`:

```python
HANDOFF_SUMMARY_MAX = 4096
HANDOFF_LIST_ITEM_MAX = 512
HANDOFF_LIST_MAX_ITEMS = 20

# Role constant — single source of truth for cross-repo handoff enforcement
CROSS_REPO_HANDOFF_ROLE = "PRESIDENT_GLOBAL"

# Pydantic annotated type for bounded string lists
BoundedStrList = Annotated[
    list[Annotated[str, Field(max_length=HANDOFF_LIST_ITEM_MAX)]],
    Field(max_length=HANDOFF_LIST_MAX_ITEMS),
]

class HandoffPacket(BaseModel):
    source_repo: str
    target_repo: str
    summary: str = Field(max_length=HANDOFF_SUMMARY_MAX)
    question: str = ""
    decisions: BoundedStrList = Field(default_factory=list)
    artifacts: BoundedStrList = Field(default_factory=list)
    open_risks: BoundedStrList = Field(default_factory=list)
    related_session_ids: list[str] = Field(default_factory=list, max_length=HANDOFF_LIST_MAX_ITEMS)
```

Add a validation helper:

```python
def validate_handoff(payload: dict[str, Any]) -> HandoffPacket | None:
    """Validate handoff sub-object if present. Returns parsed packet or None."""
    raw = payload.get("handoff")
    if raw is None:
        return None
    return HandoffPacket(**raw)  # raises ValidationError on bad input
```

**Pydantic enforcement**: List length and item size are enforced by Pydantic's `Annotated` + `Field(max_length=...)` at the type level. No separate validator needed.

**Security note on source_repo**: In this system, all callers are authenticated workers or operator tools (meshctl). There is no untrusted user input. However, to prevent misconfiguration, the router cross-checks: if `handoff.target_repo` differs from the step's `repo` field, the request is rejected (400). The step `repo` field determines actual routing; `handoff.target_repo` is informational metadata that must match it.

### 20B — Role + Topology Enforcement in add_step()

Modify `add_step()` in `src/router/thread.py`:

1. After parsing `ThreadStepRequest`, call `validate_handoff(step_request.payload)`
2. If handoff is present, cross-check `handoff.target_repo == step_request.repo` (or step_request.repo is empty). Mismatch → 400.
3. If handoff is present and `source_repo != target_repo`:
   - Check that `step_request.role == CROSS_REPO_HANDOFF_ROLE` (uses constant from models.py)
   - If not: raise error → server returns **403 Forbidden**
4. If topology is loaded:
   - Validate `handoff.target_repo` exists in topology repos
   - Validate `handoff.source_repo` exists in topology repos (unless empty string)
   - If invalid: raise error → server returns **400 Bad Request**
5. Same-repo handoffs (`source_repo == target_repo`) skip role enforcement

Server-side error handling in `_handle_add_step()`:
- Catch `pydantic.ValidationError` from handoff validation → 400
- Catch custom `HandoffRoleError` → 403
- Catch custom `HandoffRepoError` → 400

### 20C — meshctl Visibility

Extend `meshctl thread status`:
- In the step table, append `[HANDOFF]` marker to the TITLE column for steps with `payload.handoff`
- This requires the server to include a `has_handoff: bool` field in thread status step responses

Add `meshctl thread handoff <thread_id_or_name> <step_index>`:
- Fetches step task by thread + step_index
- Reads `task.payload.handoff` from a new endpoint or existing task endpoint
- Displays formatted handoff details:
  ```
  HANDOFF: backend → platform
  Summary: <text>
  Question: <text>
  Decisions: (3)
    - decision 1
    - decision 2
    - decision 3
  Artifacts: (1)
    - artifact 1
  Open Risks: (2)
    - risk 1
    - risk 2
  Related Sessions: sess-abc, sess-def
  ```

Server support:
- `GET /threads/<id>/status` — add `has_handoff` bool to step dicts
- `GET /tasks/<task_id>` already returns full task with payload — reuse for handoff details

## Acceptance Criteria

1. A thread step with `payload.handoff` containing valid fields is accepted
2. A malformed handoff payload returns 400 with descriptive error
3. A cross-repo handoff without PRESIDENT_GLOBAL role returns 403
4. `meshctl thread status` shows `[HANDOFF]` on relevant steps
5. `meshctl thread handoff <id> <step>` displays full handoff details
6. Existing thread/task behavior is unchanged for steps without handoff
