**Create comms.py module:**
Define the communication hierarchy enforcement:

```python
# Allowed communication edges (directed graph)
HIERARCHY_EDGES = {
    CommunicationRole.BOSS: {CommunicationRole.PRESIDENT},
    CommunicationRole.PRESIDENT: {CommunicationRole.WORKER, CommunicationRole.BOSS},
    CommunicationRole.WORKER: {CommunicationRole.PRESIDENT},
}
```

Implement `CommunicationPolicy` class with these methods:
1. `can_create_task(creator_role: str) -> bool` — Only boss/president can create tasks. Workers cannot create tasks for other workers.
2. `can_dispatch_task(dispatcher_role: str) -> bool` — Only president role dispatches tasks to workers (scheduler acts on behalf of president).
3. `can_ack_task(worker_id: str, task: Task) -> bool` — Only the assigned worker can ack their own task (check task.assigned_worker == worker_id).
4. `can_complete_task(worker_id: str, task: Task) -> bool` — Only the assigned worker can report completion (check task.assigned_worker == worker_id).
5. `can_view_all_tasks(role: str) -> bool` — Boss/president can view all tasks. Workers have scoped visibility.
6. `validate_communication(sender_role: str, receiver_role: str) -> bool` — Check if the sender→receiver edge exists in HIERARCHY_EDGES.

Each method returns bool. On False, the caller should raise/handle the violation. Keep the module simple — pure validation functions, no side effects.

**Important:**
- Follow existing code patterns: use `str` for role values (not enum objects) in model fields, matching the pattern used by `TaskStatus` and `CLIType`
- Do NOT modify any existing function signatures — this is additive only
- The `CommunicationPolicy` class should be stateless (no DB dependency) — it's a pure policy engine
