# Plan 16-01: Aggregator + Error Handling

## Goal

Implementare per-step `on_failure` policy (abort/skip/retry), migliorare `meshctl thread status` con audit trail completo, e creare E2E test cross-repo 3-step.

## Success Criteria (from ROADMAP)

1. Thread 3-step cross-repo esegue E2E senza copia-incolla
2. Step fallito con `on_failure: retry` viene ri-eseguito (max 3 tentativi)
3. Step fallito con `on_failure: skip` non blocca thread
4. `meshctl thread status` mostra tabella leggibile con risultati per step
5. Audit trail completo in DB: ogni step ha input, output, timestamps, worker, repo

### Task 4: Scheduler — Retry policy integration in report_failure

**Files**: `src/router/scheduler.py`

**Changes**:

In `report_failure()`:
1. Read `task.on_failure` (new field)
2. If `on_failure == "retry"`:
   - Check if `task.attempt < 3` (max_attempts)
   - If retryable: **DO NOT** transition to failed. Instead, requeue:
     - Use `requeue_task()` from heartbeat.py (same mechanism as stale sweep)
     - Set `not_before` via RetryPolicy.calculate_not_before() for backoff
     - Insert event `step_retry_requeued`
     - Update thread status (stays `active` since task is queued, not failed)
     - **Return True** without calling `on_task_terminal` (task is not terminal)
   - If NOT retryable (attempts exhausted): proceed with normal failure flow (transition to failed, call on_task_terminal)
3. If `on_failure == "abort"` or `"skip"`: proceed with existing flow (both result in the task being marked failed, which is terminal)

**Tests**: test retry requeues task, test retry exhaustion falls through to failure.

## Risks

- **Retry + transaction**: `report_failure()` currently runs in a single transaction. Adding retry logic means we need to requeue INSTEAD of failing, within the same transaction. Using `requeue_task()` which already handles this.
- **Crash during retry**: If router crashes after marking task failed but before requeue, `resolve_blocked_tasks()` recovery fallback handles it. The `not_before` field is persisted in DB, so backoff survives crash.
- **Skip + context contract**: Workers receiving `thread_context` with skipped markers must handle `result: null`. This is documented in the API response.
