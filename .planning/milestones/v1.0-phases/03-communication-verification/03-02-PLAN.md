<objective>
Implement mandatory verifier gate for critical tasks with rejection workflow and escalation.

Purpose: Critical tasks (flagged by BOSS/PRESIDENT) must pass through a review state before completion. The PRESIDENT acts as verifier — approving or rejecting work. Rejection creates a fix task that blocks re-approval. After 3 rejections, escalation to BOSS. This is a non-negotiable requirement (VERIFY-01).
Output: `verifier.py` module with approval/rejection workflow, scheduler integration, and comprehensive tests.

**Integration note:** This phase builds internal Python modules. The `check_review_timeout()` function follows the same pattern as `run_stale_sweep()` in heartbeat.py — it's a callable function that will be wired into the periodic event loop in Phase 5 (Deployment). HTTP API for PRESIDENT approval/rejection comes in Phase 4 (Event Bridge).

**Create verifier.py module:**

Implement `VerifierGate` class:

1. `should_review(task: Task) -> bool`:
   - Returns `task.critical == True`
   - Simple check used by scheduler to decide running→review vs running→completed

2. `has_pending_fixes(db: RouterDB, task_id: str) -> bool`:
   - Query tasks where `parent_task_id == task_id` AND status NOT in terminal states (completed, failed, timeout, canceled)
   - Returns True if any non-terminal fix tasks exist
   - Used by approve to prevent premature approval

3. `approve_task(db: RouterDB, task_id: str, verifier_id: str) -> TransitionResult`:
   - Check `has_pending_fixes()` — if True, return failure ("pending fix tasks")
   - Use `apply_transition()` from fsm.py: review → completed
   - Log approval event with verifier_id in payload
   - Return TransitionResult

4. `reject_task(db: RouterDB, task_id: str, verifier_id: str, reason: str) -> Task`:
   - Increment `rejection_count` on original task (direct DB update)
   - Check escalation: if `rejection_count >= 3`, emit escalation event (reuse `EscalationCallback` protocol from retry.py pattern) and return without creating fix task
   - Create new fix Task:
     - `parent_task_id = task_id` (links fix to original)
     - `title = f"Fix: {original_task.title} (rejection #{rejection_count})"`
     - `critical = False` (fix tasks don't need re-verification themselves)
     - `created_by = verifier_id`
     - `target_cli = original_task.target_cli`
     - `target_account = original_task.target_account`
     - `payload = reason` (rejection reason as context for the fix)
   - Log rejection event with reason and fix_task_id in payload
   - Return the fix task

5. `check_review_timeout(db: RouterDB) -> list[str]`:
   - Find tasks in `review` state where `review_timeout_at < now()`
   - For each: emit timeout event, transition review → timeout (existing FSM path)
   - Return list of timed-out task_ids
   - Called periodically (like stale sweep in heartbeat.py)

**Important:**
- Reuse existing patterns: `apply_transition()` for FSM transitions, `db.transaction()` for atomicity
- The fix task is a regular task in the system — it gets queued, dispatched, executed like any other
- Original task stays in `review` throughout — no new FSM states needed
- `rejection_count` is on the ORIGINAL task, not the fix tasks
- Follow the EscalationCallback protocol pattern from retry.py for escalation hooks
