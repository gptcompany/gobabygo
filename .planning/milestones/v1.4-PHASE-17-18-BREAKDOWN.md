# v1.4 Phase 17+18 Breakdown

**Purpose:** Minimal-diff implementation plan for the first two v1.4 phases.

## Architectural Correction

The current session bus is not acceptable as the primary interactive control path.

Why:

- `src/router/session_worker.py` sends inbound content with `_tmux_send_text()`
- `_tmux_send_text()` injects text line-by-line and always presses `Enter`
- empty messages are skipped

Result:

- good for audit, output stream, and quick text
- not good for raw keys, TUI navigation, or primary terminal control

v1.4 therefore keeps the bus as a control-plane only and moves the real interactive data-plane to `upterm` or `ssh+tmux`.

## Phase 17: Role Topology + Manual Wiring

### Reuse First

- Keep current env-based runtime deployment
- Do not change FSM
- Do not change DB schema for v1.4 Phase 17

### Required Scheduler Change

Phase 17 must make the scheduler topology-aware.

Current state:

- `Task.repo` already exists and is persisted
- `Scheduler.find_all_eligible_workers()` only filters by `target_cli`, `target_account`, and `execution_mode`
- `Worker` has no `repo` field

Correct v1.4 change:

- do **not** add `repo` to `Worker`
- load topology mapping `repo -> worker_pool`
- apply current eligibility filters first
- if `task.repo` is set and topology provides a pool, intersect eligible workers with that worker pool
- if no pool is defined, preserve legacy behavior

### Minimal Files To Touch

- `src/router/scheduler.py`
  - make `find_all_eligible_workers()` topology-aware using `repo -> worker_pool`
- `deploy/topology.v1.4.example.yml`
  - new seed schema for manual routing and hierarchy
- `src/meshctl.py`
  - add read-only topology inspection or validation command if needed
- `tests/router/test_scheduler.py`
  - add repo-affinity worker selection tests
- `tests/test_meshctl.py`
  - validate read-only topology command output if implemented

### Files To Avoid Touching

- `src/router/db.py`
- `src/router/fsm.py`
- `src/router/dependency.py`
- `src/router/verifier.py`

### Practical Output

Phase 17 is complete when the operator can answer, from config alone:

- who is `BOSS_GLOBAL`
- who is `PRESIDENT_GLOBAL`
- which `REPO_LEAD` owns each repo
- which workers are valid for each repo
- which host each worker lives on
- which Matrix room receives repo/global alerts

And the scheduler can enforce the declared worker pool while preserving the existing task/thread model.

## Attach Metadata Contract

### Required Fields

- `attach_kind`
  - `upterm`
  - `ssh_tmux`
- `attach_target`
  - canonical machine-readable target string

### Existing Fields To Preserve

- `tmux_session`
- `working_dir`
- `task_title`

### Exact `attach_target` Format

- For `attach_kind=upterm`
  - `ssh://TOKEN@HOST:PORT`

- For `attach_kind=ssh_tmux`
  - `ssh://USER@HOST:22?tmux_session=SESSION_NAME`

### Rendering Rule

- Router stores the canonical `attach_target`
- `meshctl` and Matrix render a human-friendly attach command from that target
- The stored field remains stable and parseable

## Upterm Lifecycle (Phase 18 Default)

### Ownership

- `session_worker` owns the `upterm` process lifecycle
- router stores attach metadata only
- router does not store SSH key material and does not manage `upterm` processes

### Start Sequence

1. `_tmux_new_session()` creates the tmux session
2. `_open_session()` persists the session record
3. `session_worker` starts `upterm host` as a child process for that specific tmux session
4. `upterm` uses a forced command that attaches to the existing tmux session
5. `session_worker` parses the emitted SSH command/target and writes canonical attach metadata into `Session.metadata`

### Runtime Contract

- default mode is per-session `upterm`
- one attach endpoint per interactive session
- the attach target is opaque to the router; the router only stores and returns it
- if `upterm` startup fails, `attach_kind` falls back to `ssh_tmux` when configured, otherwise attach is marked unavailable and the task still runs

### Stop Sequence

1. tmux session exits or task is force-closed / times out
2. `session_worker` closes the router session state as it already does
3. `session_worker` terminates the corresponding `upterm` child process
4. if graceful termination fails, `session_worker` kills the child process

### Security Mode

- v1.4 initial rollout does **not** extend `/register`
- v1.4 initial rollout does **not** add SSH key fields to `Worker`
- base mode can use `upterm` default session string semantics
- hardened mode, if needed later, is a local `session_worker` policy (`--authorized-keys` or equivalent), not a router registration concern

## Fastest Implementation Path

1. Add config knobs to `session_worker` for attach mode selection
2. Start per-session `upterm` immediately after `_open_session()`
3. Generate attach metadata from the resulting `upterm` target
4. Persist attach metadata by reusing existing `Session.metadata`
5. Expose it through `meshctl`
6. Do not modify task lifecycle or session bus semantics in Phase 17+18 beyond topology-aware worker selection

## Deferred On Purpose

- no raw-key custom transport
- no new PTY protocol
- no OpenMemory write path yet
- no handoff object yet
- no Matrix bridge yet
