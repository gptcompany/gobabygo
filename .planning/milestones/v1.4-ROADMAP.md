# Roadmap: v1.4 Native Cross-Repo Handoff

**Status:** PLANNED
**Defined:** 2026-03-04
**Execution order:** 17 -> 18 -> 19 -> 20 -> 21

## Scope

v1.4 closes the operational gaps left after v1.3:

- stop using the current `/sessions/*` HTTP bus as if it were a full terminal transport
- preserve cross-repo context without copy/paste
- make interactive sessions reachable from Android without desktop sharing
- formalize the cross-repo hierarchy and routing model
- add a shared memory layer without replacing the router as the source of truth

## Operating Model

### Planes

- `gobabygo` router/DB = operational truth (`task`, `thread`, `session`, state, audit)
- session bus (`/sessions/*`) = control-plane only (audit, output stream, quick text)
- `upterm` / `ssh+tmux` = primary interactive data-plane
- `Matrix` = notification plane
- `OpenMemory` via MCP = shared memory plane

### Hierarchy

- `BOSS_GLOBAL` = priorities, approvals, escalation
- `PRESIDENT_GLOBAL` = cross-repo coordinator and owner of handoff graph
- `REPO_LEAD` = repo-scoped tactical coordinator
- `WORKER` = execution only
- `OPERATOR` = human attach/intervention actor

### Communication Rules

- `BOSS_GLOBAL <-> PRESIDENT_GLOBAL`
- `PRESIDENT_GLOBAL <-> REPO_LEAD`
- `REPO_LEAD <-> WORKER`
- `PRESIDENT_GLOBAL -> REPO_LEAD(target_repo)` for inter-repo handoff
- `WORKER -> WORKER` forbidden
- `REPO_LEAD(repo A) -> REPO_LEAD(repo B)` forbidden in v1.4 initial rollout

### Configuration Model

- Per-host runtime configuration stays env-based (`MESH_ROUTER_URL`, `MESH_WORKER_ID`, `MESH_CLI_TYPE`, `MESH_ACCOUNT_PROFILE`, `MESH_EXECUTION_MODES`, `MESH_CLI_COMMAND`)
- Cross-repo orchestration adds one manual topology file (`deploy/topology.v1.4.example.yml` as seed schema)
- The topology file complements env files; it does not replace them
- `Task.repo` remains the task-side routing hint; v1.4 uses topology-defined `repo -> worker_pool` constraints instead of adding `repo` to the `Worker` model

## Phase 17: Role Topology + Manual Wiring

**Goal:** Make the operational hierarchy explicit before adding more transport or memory features.
**Depends on:** v1.3 shipped
**Requirements:** TOPO-01, TOPO-02, TOPO-03, TOPO-04, TOPO-05
**Success Criteria:**
1. A manual topology file defines `global`, `hosts`, `workers`, and `repos`
2. Every active repo resolves deterministically to one `REPO_LEAD`, one preferred host, and a bounded worker pool
3. Cross-repo routing policy is explicit (`require_president_handoff=true` in the initial rollout)
4. Scheduler candidate selection becomes topology-aware: when `task.repo` is set and a repo worker pool exists, eligible workers are restricted to that pool after the existing `cli/account/mode` filters
5. No FSM or DB schema changes are required to declare topology

## Phase 18: Native Session Attach

**Goal:** Give every interactive session a real attach handle without changing the tmux runtime model.
**Depends on:** Phase 17
**Requirements:** ATCH-01, ATCH-02, ATCH-03, ATCH-04
**Success Criteria:**
1. `session_worker` publishes an attach handle for every `execution_mode=session` task
2. `Session.metadata` includes `attach_kind` and `attach_target` in addition to existing `tmux_session`, `working_dir`, and `task_title`
3. `meshctl` can display attach information without direct DB access
4. The current `/sessions/*` bus remains unchanged as audit/control-plane only

Plans:
- [ ] 18-01-PLAN.md -- Upterm/ssh attach handle generation, metadata enrichment, meshctl read path, tests

## Phase 19: Approval + Notification Bridge

**Goal:** Notify the operator quickly when a human needs to intervene, and include the correct attach handle.
**Depends on:** Phase 18
**Requirements:** NOTIFY-01, NOTIFY-02, NOTIFY-03, NOTIFY-04
**Success Criteria:**
1. Matrix messages are emitted for `approval_needed`, `input_requested`, `thread_blocked`, `thread_failed`, and `thread_completed`
2. Each notification includes repo, task/session/thread identity, reason, and attach handle
3. Quick text replies remain optional fallback only; they do not replace terminal attach
4. The bridge runs out-of-process (ideal host: `muletto`) and does not become part of the router critical path

Plans:
- [ ] 19-01-PLAN.md -- Matrix bridge, trigger mapping, message format, operator workflow docs

## Phase 20: Structured Cross-Repo Handoff

**Goal:** Replace copy/paste with a structured handoff object while reusing the existing thread model.
**Depends on:** Phase 19
**Requirements:** HANDOFF-01, HANDOFF-02, HANDOFF-03, HANDOFF-04, HANDOFF-05
**Success Criteria:**
1. A step can carry a structured `handoff` payload with summary, question, decisions, risks, artifacts, and related sessions
2. `PRESIDENT_GLOBAL` is the only cross-repo handoff producer/consumer in the initial rollout
3. `thread_context` stays available, but handoff carries the semantic cross-repo packet
4. The first iteration reuses existing payload/event structures; no second orchestration stack is introduced

Plans:
- [ ] 20-01-PLAN.md -- Handoff shape, server/API wiring, meshctl visibility, tests

## Phase 21: Shared Memory Layer (OpenMemory via MCP)

**Goal:** Add organization-wide shared memory without changing operational authority.
**Depends on:** Phase 20
**Requirements:** MEM-01, MEM-02, MEM-03, MEM-04
**Success Criteria:**
1. OpenMemory runs as a separate service and is reachable through its native MCP server
2. Memory entries cover decisions, closed-thread summaries, recurring issues, and repo dependency context
3. Memory writes are best-effort; router dispatch/ack/complete never depend on OpenMemory availability
4. `OpenMemory` remains a memory sidecar only, not a message bus or task router

Plans:
- [ ] 21-01-PLAN.md -- OpenMemory deployment mode, MCP access pattern, memory write/read policy, docs

## Non-Goals

- Replace `tmux` with a custom PTY protocol
- Use the current `/sessions/*` bus as the primary interactive transport
- Turn Matrix into a full terminal
- Introduce worker-to-worker direct communication
- Replace router/DB with OpenMemory

## Acceptance Summary

v1.4 is successful when:

1. Interactive sessions are reachable by attach handle, not by manual hunt through tmux panes
2. Cross-repo work no longer depends on copy/paste as the primary handoff mechanism
3. The current session bus is explicitly treated as a limited control-plane
4. Shared memory exists, but operational authority remains in `gobabygo`
