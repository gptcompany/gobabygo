# v1.4 Phase 19 Breakdown

**Purpose:** Define the first implementation of the Matrix notification bridge with minimal coupling and maximum reuse.

## Architectural Position

Phase 19 is a notification-plane feature only.

It must not:

- become a second source of truth
- become a task router
- replace terminal attach
- depend on direct DB access

The bridge is intentionally out-of-process and should run on `muletto` in the initial rollout.

## Reuse First

- Reuse existing router HTTP API only
- Reuse `Session.metadata.attach_kind` / `attach_target` from Phase 18
- Reuse existing `GET /sessions`, `GET /sessions/<id>`, and `GET /sessions/messages`
- Reuse existing task and thread read paths
- Keep Matrix bridge completely separate from router and workers

## MVP Consumption Model

The bridge polls the router.

Initial read sources:

- `GET /sessions?state=open`
- `GET /sessions/messages?session_id=...&after_seq=N`
- `GET /tasks?status=review`
- `GET /threads`

No direct DB reads in the MVP.

If later needed, add a dedicated event stream endpoint, but do not block Phase 19 on that.

## Trigger Rules (MVP)

### `input_requested`

Raised when a new CLI output message on an open session matches configured human-input patterns.

Examples:

- `approve`
- `continue`
- `press enter`
- `y/n`
- `select`

This is a best-effort heuristic and should remain configurable.

### `approval_needed`

Raised when a task enters `review`.

### `thread_blocked`

Raised when a thread status changes to `blocked`.

### `thread_failed`

Raised when a thread status changes to `failed`.

### `thread_completed`

Raised when a thread status changes to `completed`.

## State Tracking

The bridge needs only lightweight local state:

- last seen open sessions
- last seen message sequence per session
- last seen task ids in `review`
- last seen thread status by `thread_id`

This state lives in the bridge process and is rebuildable.

No new router tables are required for Phase 19.

## Message Contract

Each Matrix notification should include:

- repo
- session id (if any)
- task id (if any)
- thread id (if any)
- reason / trigger
- attach kind
- attach target
- a rendered human-friendly attach command

Optional:

- short excerpt of the triggering CLI message
- quick reply hint (`text reply possible, full control requires attach`)

## Attach Coupling Rule

The Matrix bridge consumes the canonical `attach_target` from Phase 18.

Rules:

- router stores canonical machine-readable attach target
- Matrix bridge renders user-facing copy from it
- if attach target format changes later, only renderers change; router storage stays canonical

## Human Control Policy

- Matrix quick replies are allowed only for simple text responses
- terminal attach remains the primary path for real interaction
- the bridge must state this explicitly in the notification copy when relevant

## Minimal Files To Touch (when implemented)

- new external process/script for Matrix bridge (preferred outside `src/router/`)
- `deploy/` docs and service notes
- optionally `src/meshctl.py` only if a local inspection helper materially improves operator workflows

## Files To Avoid Touching

- `src/router/fsm.py`
- `src/router/scheduler.py`
- `src/router/dependency.py`
- `src/router/verifier.py`
- `src/router/db.py`

Router changes should be avoided in the MVP unless a missing read endpoint blocks the bridge.

## Acceptance

Phase 19 is complete when:

1. An open interactive session that requests input produces a Matrix notification
2. The notification includes the correct attach handle
3. A thread status transition to `failed` or `completed` produces a single useful message
4. The bridge can fail or restart without breaking task execution
