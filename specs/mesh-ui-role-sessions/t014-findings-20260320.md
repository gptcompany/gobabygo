# Mesh UI Role Sessions T014 Findings — 2026-03-20

## Current Verdict

- `T001` through `T013` are implemented.
- `T014` is still open.
- The remaining blockers are no longer basic design gaps; they are live integration and control-plane correctness issues.
- This file is the handoff package for external review before further implementation.

- The operator Mac can drive iTerm2.
  - A direct Python+iTerm2 smoke created tabs and injected text successfully.
- `mesh ui` no longer depends on raw CLI launch.
  - The current path creates task-backed role sessions through the router.
- Gemini works on WS `.111` for `sam`.
  - Direct `ccs gemini` works from `sam@5610`.
- Concurrent Gemini sessions under `sam` are viable.
  - Two independent `tmux` sessions running `ccs gemini` stayed alive for about two minutes in both `/tmp/mesh-tasks/snake-game` and `/media/sam/1TB/snake-game`.
- After the latest router/runtime fixes, backend role sessions now materialize concurrently.
  - For `ui_group_id = snake-game-ui-20260320T110208Z`, the router showed:
    - `president`: `running`, open session `e67615f8-5ad0-4d8a-a329-45a00fb9d6f9`
    - `lead`: `running`, open session `a0320a2d-fff0-4d77-80d4-6cf3aa67340c`

## Findings Closed During This Week

1. Live router on `.100` was stale and still dropped top-level `repo` / `role` on `POST /tasks`.
   - Result: UI role tasks were created without the metadata needed for role-session orchestration.
   - Fix: live router rebuilt from current code.
   - Related implementation: `01da810` `Accept repo and role on task creation`

2. Remote pane bootstrap helper path was broken.
   - Result: remote panes could start without the helper functions needed to label and banner the pane correctly.
   - Fix: `f2e7715` `Fix remote mesh ui pane bootstrap helpers`

3. Worker registration conflict handling hid real `account_in_use` failures.
   - Result: router/worker registration state looked healthy while Gemini workers were actually conflicting.
   - Fix: `01d67be` `Fail worker registration on account conflicts`

4. Extra Gemini session workers were running under the wrong user.
   - Result: `sam` could run `ccs gemini`, but extra systemd workers were still running as `mesh-worker` and could not use the valid Gemini runtime.
   - Fix: `87213e5` `Apply runtime user overrides to all session workers`

5. Router accept backlog was too small for the current long-poll + heartbeat + send mix.
   - Result: control-plane requests timed out under multi-role spawn.
   - Fix: `3a9f077` `Increase router HTTP accept backlog`

6. Provider override broke worker-pane attach scoring.
   - Result: `worker-codex` sessions created under `MESH_UI_PROVIDER_OVERRIDE=gemini` could be opened live but were still rejected by the attach resolver, so the UI waited for a nonexistent future session.
   - Fix: `815e256` `Fix mesh UI worker attach under provider override`

7. Live state was repeatedly contaminated by stale `snake-game` UI tasks and sessions.
   - Result: reruns were nondeterministic.
   - Operational remediation: direct router DB cleanup of only the `snake-game` UI-role state, then restart of the Gemini session workers.
   - Observed reset result:
     - `19` sessions closed
     - `33` UI tasks touched
     - Gemini session workers reset to idle before rerun

## Current Unresolved Findings

### 1. Likely bootstrap race in `session_worker` drops early operator input

Evidence:
- From WS `.111`, operator input was sent to `president` after the session was open:
  - `/media/sam/1TB/gobabygo/scripts/mesh send president "reply with exactly PONG"`
  - `/media/sam/1TB/gobabygo/scripts/mesh enter president`
- Router DB for session `e67615f8-5ad0-4d8a-a329-45a00fb9d6f9` contains the operator message:
  - `seq = 3016`
  - `direction = in`
  - `role = operator`
  - `content = reply with exactly PONG`
- The session did not process the message.

Likely cause:
- `src/router/session_worker.py` reads `/sessions/messages?after_seq=0` during bootstrap and then sets `after_seq` to the max seen sequence.
- If operator input arrives between session creation and that initial sync, it can be marked as already seen without ever being delivered to tmux.

Why this matters:
- Even when sessions finally materialize, role-to-role and operator-to-role messaging is not trustworthy.
- This blocks the core value of the cockpit.

### 2. Router/control-plane instability still affects multi-role runs

Observed symptoms:
- router `.100` went `unhealthy` during earlier reruns
- workers logged timeouts against:
  - `/heartbeat`
  - `/tasks/next`
  - `/sessions/send`
- stale open sessions and stale busy workers were left behind after failed runs

Current state:
- backlog tuning improved the situation enough to get concurrent backend role sessions to materialize
- but the control plane is still not proven stable under repeated T014 loops

Why this matters:
- T014 acceptance depends on deterministic live behavior, not one lucky run.

### 3. `mesh_session_cli.py list --all --json` is not reliable enough for live diagnostics in this flow

Observed behavior:
- On WS `.111`, `mesh_session_cli.py list --all --json` returned `[]` while the router DB still contained running `snake-game` tasks and open sessions.

Why this matters:
- It makes operator-side triage misleading during T014.
- The current debugging loop had to fall back to direct router DB inspection to understand the actual state.

### 4. Automated iTerm2 pane inspection is not reliable enough to serve as final proof

Observed behavior:
- `osascript` could confirm window/tab/session counts changing.
- Automated pane-content extraction was inconsistent.
- Direct iTerm2 Python introspection from the smoke environment could hang or fail to connect.

Why this matters:
- Backend state can look correct while the operator-visible cockpit still appears failed.
- Final T014 acceptance still needs either:
  - a more reliable pane-observation harness, or
  - a human-driven acceptance pass with explicit checks.

## What T014 Still Needs Before It Can Be Marked Done

1. Fix or disprove the bootstrap-race hypothesis in `src/router/session_worker.py`.
2. Re-run a multi-role cockpit and verify that early operator input is delivered to the intended live session.
3. Validate operator-visible panes, not just backend sessions.
4. Re-check:
   - `mesh send`
   - `mesh enter`
   - `mesh interrupt`
   - completion summary emission and inspection
   - `mesh ui close`
5. Confirm that rerunning the cockpit does not cross-attach to stale groups.

## Recommended External Review Focus

1. `src/router/session_worker.py`
   - review the initial message sync after session open
   - verify whether operator messages can be lost during bootstrap

2. `src/router/server.py` and live router behavior
   - review whether the current threaded HTTP server and backlog are sufficient for the observed request mix
   - focus on long-poll, heartbeat, ack, and message-send concurrency

3. `scripts/mesh_iterm_ui.py`
   - verify attach-or-spawn behavior now that provider override and live-session scoring were patched
   - look for remaining races between pending-task reuse, attach resolution, and pane dispatch

4. `scripts/mesh_session_cli.py`
   - review live listing and group resolution behavior under stale/open/failed mixed states
   - confirm whether `list --all` can legitimately hide live sessions in this path
