# Mesh UI Role Sessions T014 Handoff — 2026-03-23

## Executive Summary

`T014` is still open.

The mesh control plane is no longer the main blocker. The main blocker at this
checkpoint is the Mac-side `iTerm2` Python API. New `mesh ui` launches can hang
or fail before any UI tasks are created because `iTerm2` is not exposing its
private Python API socket on the operator Mac.

Current canonical live topology for this work:

- router: `10.0.0.4:8780` on `vpsazure`
- worker host: `.111` / `192.168.1.111`
- operator UI host: the Mac running `iTerm2`

## User Goal

The target UX is:

- the human talks to `boss` as a real AI CLI, not only an operator shell
- `boss` delegates to `president`
- `president` coordinates `lead`, `worker-gemini`, and `verifier`
- the worker roles operate on the real target repo on the WS, not in `/tmp`
- worker outputs should be concise and task-shaped:
  - summary
  - artifacts
  - commit hash if they made one

## What Is Already Fixed In Local Code

These local Mac-side changes exist in the current working tree and are covered by
tests:

1. bare repo names now resolve to the real WS repo base under `/media/sam/1TB`
   instead of defaulting to `/tmp/mesh-tasks/...`
2. `boss` is now modeled as a provider-backed Gemini role in
   `mapping/operator_ui.yaml`
3. `boss` bootstrap prompt now explains that `president` is the execution
   coordinator and that routed communication should use `mesh send`
4. `mesh_ui_role_shell.sh` exports the `mesh` script directory into `PATH` so
   role panes can actually call `mesh send`
5. live attach fallback for session lookup is more robust against slow router
   task/session detail paths
6. router env lookup now prefers `/etc/mesh-worker/common.env` over stale
   `~/.mesh/router.env`
7. duplicate initial bootstrap prompts were fixed in `session_worker.py`
8. `mesh_iterm_ui.py` now fails fast with a clear message when the `iTerm2`
   Python API socket is missing, instead of hanging forever in retry mode

## Current Hard Blocker

The current blocker is the Mac app state for `iTerm2`, not the mesh router.

Observed directly on the Mac:

- `~/Library/Application Support/iTerm2/private/socket` is missing
- direct `iterm2` Python API probe fails with:
  - `There was a problem connecting to iTerm2`
- new `mesh ui` runs therefore stop before creating any router tasks

This explains why fresh `snake-game` launches wrote a cache file but created no
new tasks or sessions on the router.
